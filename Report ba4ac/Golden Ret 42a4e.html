Golden Retriever Guide (1)

Introduction

The Golden Retriever was created by the AISG team to help them handle email queries. After seeing the model being successful in helping them to deal with the queries, they decided to make the Golden Retriever open-sourced so that other companies who need an information retrieval engine for human language queries can use it for their own usage.

Purpose

The main purpose of us using the Golden Retriever is to

  • Explore other options available to us.
  • Use it as our base model
  • Compare it with our current model, Haystack to show why and how Haystacks is a better option
  • To satisfy our stakeholders‚Äô needs

Guide to using the Golden Retriever

1. Check if you had docker install by running `docker -v && docker-compose -v`

2. Set up a virtual environment by running `virtualenv env` or `python -m venv env`

3. To open and use the virutal environment run `source env/Scripts/activate`

4. Install required packages by running `pip install -r requirements.txt`

5. Create a .env file in the project root folder to manage Minio access keys. Example:

`

MINIO_URL=localhost:9001

ACCESS_KEY=minio # change this to a proper access key!

SECRET_KEY=minio123 # change to to a proper secret key!

`

5.1) Change the docker-compose.yml file, change the access key and secret key to your own settings.

5.2) When encountering a ‚Äú elasticsearch: [1]: max virtual memory areas vm.max_map_count [65530] is too low, increase to at least [262144] ‚Äù error ,use WSL system

Run this code on powershell: wsl -d docker-desktop

This is in order to activate the subsystem

Then use this code to change the memory count

sysctl -w vm.max_map_count=262144

5.3) At minio_handler.py , remove all the ResponseError, BucketAlreadyOwnedByYou, BucketAlreadyExists inside the file and change it all to just 1 error called InvalidResponseError

6. Build and start up the multi-container application by running `docker-compose -f docker-compose.yml build` and `docker-compose -f docker-compose.yml up`

6.1) The app is suppose to exit where as minio and elk is suppose to be runnning

7) Add -- in front of url, csv file and index_name at create_doc_index.py, at parser.add_arguements

7.1) Csv columns have to follow this sequence

ans_id

ans_str

query_id

query_str

7.2) Upload your datasets by running `python -m src.elasticsearch.create_doc_index --url localhost --csv_file data/(yourcsvfilename).csv --index_name (yourownindexname)`

  • -url: elasticsearch url
  • -csv_file: datasets (csv file with QA pairs)
  • -index_name: name of index to create (e.g: test)

8. Amend setup configuration in `app/api/config.py`

  • -QA_INDEX: index name that you created (e.g: test, (yourownindexname))
  • -INDEX_BUCKET = 'test-index'
  • -INDEX_PICKLE = 'test.pkl'
  • -INDEX_FILE = 'test.idx'
  • -INDEX_PREFIX = 'test'

8.1) If u see an ‚ÄúOSError Saved Model does not exist‚Äù error, at logs, click into your tfhub_modules there and enter it to empty the folder to clear cache.

9.Amend setup configuration in `app/api/config.py`.

10. To run the finetune scripts run `python -m src.finetune.main` #you can amend src/finetune/config.py to insert preferred finetuning parameters

11. To encode your documents `python -m src.dvc_pipeline_scripts.index_encode --data ./data/test.csv --output_folder model_artefacts --index_prefix test --gr_model USEEncoder --savedir ./test_finetune/model_test/USE/best/0/`

  • -data: path to file with raw responses
  • -output_folder: path to save index
  • -index_prefix: prefix to add to -data.pkl of index
  • -gr_model: name of gr model
  • -savedir: path to fine-tuned encoder weights
  • -index_name the index name define earlier when uploading datasets(e.g:test)

12. To start up backend app run `uvicorn app.api.main_es:app`

13. Start the front end using: streamlit run app/streamlit/main.py -- --url http://localhost:8000

Note: Use streamlit version 0.60.0
